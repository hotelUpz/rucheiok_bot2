# Rucheiok Bot 2.0 🎯 | Phemex HFT Sniper Bot

**Rucheiok Bot 2.0** — это высокочастотный (HFT) асинхронный торговый бот для фьючерсного рынка **Phemex USDT Futures**. 
Архитектура бота построена на принципах state-driven разработки, строгой изоляции торговых потоков (Hedge-mode ready) и бескомпромиссной реактивности через WebSocket. Бот предназначен для точечной работы с ликвидностью: он заходит в позицию строго лимитными ордерами по паттернам стакана и использует динамическую логику выхода ("Объемный Снайпер").

## 🚀 Ключевые особенности
* **Абсолютная асинхронность:** Построен на `asyncio` и `aiohttp`, выдерживает массированный поток стаканов (Orderbook Depth) по десяткам символов без блокировок.
* **Объемный Снайпер (Dynamic Limit Taker):** Бот не ставит статичные тейк-профиты. Он виртуально вычисляет цель, сканирует стакан в реальном времени и бросает лимитку ровно в тот уровень, который способен поглотить нужный объем.
* **Hedge-Mode Ready:** Архитектура композитных ключей (`BTCUSDT_LONG`, `BTCUSDT_SHORT`) позволяет независимо торговать обе стороны по одному инструменту без риска перезаписи стейта.
* **Zero-Latency Cancel:** Главное правило системы — при получении статуса `PartiallyFilled` остаток ордера отменяется фоновой задачей за миллисекунды, расчищая путь для новой итерации парсинга стакана.
* **Неуязвимый State Manager:** Локальное сохранение состояния защищено от сбоев REST API, разрывов WebSocket-соединения и "фантомных наливов". При рестарте система жестко синхронизируется с биржей, отсекая мусор.

## 🛡️ Золотые Инварианты Системы
1. По одной монете-стороне (например, `BTCUSDT_LONG`) может быть только один торговый цикл.
2. Все входы и выходы — **строго лимитные**. Никаких слепых ударов по маркету.
3. Остатки после `PartiallyFilled` (как на входе, так и на выходе) отменяются мгновенно (с учетом entry_timeout_sec и hunting_timeout_sec).
4. Внешнее/Ручное закрытие: если оператор закрыл позу руками, бот сбрасывает стейт без штрафов и карантинов. Монета снова свободна.
5. Идемпотентность: дублирующие WS-события или запоздалые наливы корректно обрабатываются без потери баланса (`pos.qty`).
6. При жестких отказах API (Reject), если позиция уже в рынке, бот переходит в режим "Экстрим", а при невозможности закрытия — вызывает оператора, **никогда не удаляя** стейт.

## 🧬 Жизненный цикл позиции (Exit Scenarios)
Бот сопровождает сделку 5-ю параллельными процессами:

1. **Average Scenario (Сердце):** Сканирует стакан в поисках максимального объема по цене `>=` (для лонга) или `<=` (для шорта) виртуального тейка. При нахождении — бьет туда. Со временем плавно снижает аппетиты (`shift_demotion`).
2. **Negative Scenario:** Мониторит стакан на предмет "топтания на месте" или ухода в минус. При затяжном негативе (хуже `entry_price`) — сбрасывает позу в Extrime Close.
3. **Interference (Скупка помех):** Обнаруживает мелкие преграды на пути к цели (аски для лонга, биды для шорта) в диапазоне от входа до цели. Выкупает их, чтобы расчистить путь цене, плюсуя объем к общей позиции.
4. **Position TTL:** Глобальный таймер жизни позиции. Двухэтапный: при истечении `position_ttl` секунд → `TRIGGER_BREAKEVEN` (переводим цель в безубыток с учётом `to_entry_orientation`). Если лимитка не закрыла позицию за `breakeven_wait_sec` → `TRIGGER_EXTRIME`.
5. **Extrime Close:** Механизм экстренного выхода. Рассчитывает справедливую ось `mid = (ask1 + bid1) / 2` и ставит лимитку с прогрессирующим смещением (пробивая стакан), гарантируя исполнение.

#### ~~4. Некорректная обработка задач (Tasks) в `main.py`~~ ✅ FIXED
**Где:** `main.py`, метод `_main`.
**Суть:** У вас используется `await asyncio.gather(*tasks)`. Если `tg_enabled = True`, в список `tasks` добавляется только один воркер — `tg_task`. Функция `gather` завершает работу, когда завершаются все переданные в нее задачи.
Если `polling_supervisor` вылетит с непредвиденной ошибкой или выйдет из цикла (например, CancelledError), `gather` завершится, скрипт выйдет из `_main`, и **торговый бот полностью отключится**, даже если торговые процессы внутри `TradingBot` работали штатно!
✅ **Как исправить:**
Вместо `gather` лучше использовать бесконечный `Event` для удержания главного цикла:
```python
stop_event = asyncio.Event()

# ... запуск бота и tg_admin ...
if tasks:
    # Запускаем задачи в фоне, не привязывая смерть бота к смерти TG
    for t in tasks:
        asyncio.create_task(t)

try:
    await stop_event.wait() # Висим тут бесконечно
except (asyncio.CancelledError, KeyboardInterrupt):
    # Штатно глушим бота
```

#### 5. Мутация состояния (State) в обход Lock-ов
**Где:** `CORE/bot.py` (`_orchestrate_market_tick`) и `EXIT/engine.py`.
В главном пайплайне `bot.py` вы вызываете:
```python
action = self.exit_engine.evaluate_pipeline(snap, pos) # БЕЗ ЛОКА
if action: 
    await self.executor.handle_exit_action(symbol, pos_key, action) # ЛОК БЕРЕТСЯ ЗДЕСЬ
```
Внутри `evaluate_pipeline` вы жестко мутируете стейт позиции (например, `pos.in_breakeven_mode = True`, `pos.current_target_rate = 0.0`, меняете таймеры).
Поскольку вы используете `asyncio` (один поток) и внутри `evaluate_pipeline` нет `await`, контекст не переключается, и гонки данных (data race) с WS формально **пока нет**.
**НО!** Это очень хрупкий дизайн (Code Smell). Если завтра вы добавите `await` внутрь `NegativeScenario` (например, логгер или запрос к БД), WS может прислать сообщение посреди выполнения пайплайна и изменить `pos.qty`, что приведет к непредсказуемому поведению.
✅ **Как исправить (Лучшая практика):**
Блокируйте позицию на весь цикл принятия решения по выходу, а не только на момент отправки ордера:
```python
async with self.executor.get_lock(pos_key):
    action = self.exit_engine.evaluate_pipeline(snap, pos)
    if action:
        await self.executor.handle_exit_action(symbol, pos_key, action)
```
*(Придется убрать взятие `lock` внутри самого `handle_exit_action`, чтобы не было Deadlock-а, либо использовать `asyncio.Condition` / реентерабельные локи, если вы используете сторонние либы).*
