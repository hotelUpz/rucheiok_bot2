## GOAL

# 1. Полное ТЗ бота, как будто проекта ещё нет

## 1.1. Назначение

Разработать асинхронного торгового бота для **Phemex Futures**, работающего по лимитной логике входа и лимитной логике выхода, с контролем позиции по стакану и ценовым условиям.

Бот должен:

* находить точку входа по паттерну стакана;
* открывать позицию **только лимитным ордером**;
* после подтверждения входа сразу переводить позицию в режим управляемого выхода;
* сопровождать позицию несколькими параллельными exit-сценариями;
* работать устойчиво при частичных исполнениях, отменах, ручных действиях оператора, реконнектах WS и рестартах процесса.

---

## 1.2. Основной режим работы

На первом этапе:

* биржа: **Phemex USDT Futures**;
* торговля: **одна активная позиция одновременно**;
* но архитектура должна сразу поддерживать расширение до `N` позиций.

Конфигурация:

* `app.max_active_positions = 1` по умолчанию;
* app.max_active_positions > 1 -- режим мультиактивов.

## 1.7. Правила торговли

### Инварианты

1. По одной монете не может быть двух независимых торговых циклов одновременно.
2. Пока позиция не завершена, монета считается занятой.
3. Все входы — только лимиткой.
4. Все выходы — только лимиткой.
5. Остаток входной лимитки после partial fill обязан быть снят. (В фоновом режиме, не блокируя выполнения остальных операций, например -- постановку закрывающей TP лимитки. Важно!)
6. После внешнего/ручного закрытия бот не “додумывает” поведение за оператора:

   * нет позиции → state сброшен;
   * монета снова свободна;
   * без штрафов, без карантина, без искусственных кулдаунов.
7. Решения по входу и выходу принимаются только по свежим данным.
8. Любое действие бота должно быть идемпотентным: повтор одного и того же WS-события не должен ломать логику.
9. Источник для входа в позицию -- только внутренние сигналы. Никаких удочерений осиротевших позиций. Благотворительностью не занимаемся и не лезем в чужие торги.

# 1.8. Подробная логика входа

## 1.8.1. Universe

Бот получает список активных USDT-фьючерсных контрактов Phemex.

Далее исключаются:

* black list;
* неактивные символы;
* символы в карантине;
* символы уже занятые позицией/ордером.

---

## 1.8.2. Entry pattern

По стакану Phemex анализируется паттерн:

* верхняя часть;
* тело;
* низ стакана;
* соотношение уровней;
* допустимая дистанция bid/ask;
* глубина анализа.

Результат:

* либо сигнала нет;
* либо сигнал есть и содержит:

  * side;
  * entry price;
  * параметры для расчёта цели.

---

## 1.8.3. Binance cross-check

Если включён Binance filter:

* сравниваются Binance и Phemex цены;
* проверяется минимальный спред;
* при необходимости используется TTL подтверждения спреда.

---

## 1.8.4. Funding filter

Если включён funding filter:

* символ блокируется перед funding-событием;
* символ блокируется, если funding превышает порог.
* (Блокируется -- жесткое слово. Просто скипается.)

Фильтр должен быть реальным gating-механизмом, а не декоративным флагом.

---

## 1.8.5. Финальное условие входа

Вход возможен только если одновременно:

* есть валидный pattern signal;
* пройден Binance-filter;
* символ прошел funding filter;
* символ не в black list;
* символ не в карантине;
* символ не занят;
* не превышен лимит активных позиций.

---

## 1.8.6. Расчёт размера входа

Размер позиции определяется по `risk`:

* `margin_size = row` или фиксированное значение; Если 'row' -- берем из первого уровня аска (для Лонг сценария) и первого уровня бида (для Шорт сценария). Суть -- выкупка объема первого уровня стакана.
* берётся объём первого уровня стакана;
* добавляется `margin_over_size_pct`; (С целью запаса на случай появления дополнительного объема в момент детерминации сигнала)
* ограничивается `notional_limit`; -- святое правило.
* переводится в количество контракта;
* округляется по `lot_size`.

---

## 1.8.7. Размещение входного ордера

Бот выставляет лимитный ENTRY order:

* side = Buy/Sell; (Лучше перейти на семантику Хедж Режима: Open Long / Open Short, Close Long / Close Short)
* pos_side = Long/Short;
* qty;
* price.

После успешной постановки:

* создаётся position state;
* сохраняется `pending_entry_order`;
* позиция считается “открывающейся”, но ещё не подтверждённой.

---

1.8.8. Обработка входного ордера по WS

Если ордер исполнен полностью — вход завершён, запускается сопровождение позиции.
Если ордер исполнен частично:
фиксируем фактически исполнённый объём;
отправляем асинхронную отмену остатка;
дальше работаем только с фактически исполнённым объёмом;
повторно отмену не дёргаем.
Если ордер отменён без исполнения:
удаляем временный state;
освобождаем символ.
Если ордер отклонён или деактивирован до входа:
удаляем временный state;
освобождаем символ;
fail/quarantine не начисляем.

---

# 1.9. Запуск exit-жизненного цикла

После подтверждения входа должны параллельно запуститься механизмы управления позицией:

* average scenario;
* negative scenario;
* interference monitor;
* global position TTL;
* extrime close engine.

Важно:

* average и negative стартуют **одновременно**;
* interference живёт параллельно;
* TTL является глобальным ограничением жизни позиции;
* extrime включается только по условиям.

---

# 1.10. Подробная логика выхода

## 1.10.1. Average scenario

Назначение: спокойно закрывать позицию в плюс по стартовой цели и затем при необходимости ухудшать цель.

Параметры:

* `stabilization_ttl`;
* `target_rate`;
* `shift_demotion`;
* `min_target_rate`;
* `shift_ttl`.

Поведение:

1. После подтверждения входа ждём `stabilization_ttl`. (0 -- не ждем)
2. После ожидания ставим первичную TP-лимитку:

   * цена определяется как доля пути от `entry_price` до `base_target_price_100`.
   sign = 1 if pos_side.upper() == "LONG" else -1
   Формула = entry_price(то есть ask1 | bid1) * (1 + sign *(spr3_pct * target_rate))
   то есть base_target_price_100 de facto == ask3 для Лонг (на момент сигнала) и bid3 для Short (на момент сигнала)

3. Далее каждые `shift_ttl`:

   * target_rate уменьшается на `shift_demotion`;
   * но не ниже `min_target_rate`.
4. После каждого сдвига:

   * negative_duration сбрасывается.

Если `stabilization_ttl == 0`, первичный TP должен ставиться сразу.

Если `average.enable == false`, этот сценарий не должен вообще управлять целью.

---

## 1.10.2. Negative scenario

Назначение: определить, что рынок после входа пошёл против позиции (либо стоит на месте), и через выдержанный негативный период перевести позицию в экстренное закрытие.

Параметры:

* `stabilization_ttl`;
* `negative_spread_pct`;
* `negative_ttl`.

Логика:

1. До истечения `stabilization_ttl` сценарий молчит.
2. Для LONG:

   * сравнивается текущий ask1 с init_ask1.
3. Для SHORT:

   * сравнивается текущий bid1 с init_bid1.
4. Если рынок находится в негативной зоне (то есть например ask1 == init_ask1 or bid1 == init_bid1 (при negative_spread_pct == 0), либо в худшую сторону):

   * идёт накопление negative_duration.
5. Если negative_duration >= negative_ttl:

   * включается extrime mode.

Если рынок перестал быть негативным:

* negative_duration обнуляется.

---

## 1.10.3. Interference

Назначение: скупка мешающих уровней между входом и текущей целью.

Параметры:

* `average_vol_pct_to_init_size`; -- если объем в пределах процента average_vol_pct_to_init_size от init_size (начально планового объема перед постановкой открывающей лимитки)
* `max_vol_pct_to_init_size`. -- суммарное значение скупок которые бот может себе позволить (как процент max_vol_pct_to_init_size от init_size (начально планового объема перед постановкой открывающей лимитки))

Логика:

* Для LONG анализируются asks между `entry_price` и ТЕЙК ПРОФИТОМ!.
* Для SHORT анализируются bids между ТЕЙК ПРОФИТОМ(!) и `entry_price`.

После выкупа помехи:

* TP должен быть переставлен на актуальный общий объём позиции (все максимально быстро). И вообще, все отмены с последующей перестановкой цели должны быть максимально быстро и эффективно.

---

## 1.10.4. Global position TTL

Назначение: позиция не должна жить бесконечно.

Параметры:

* `position_ttl`;
* `to_entry_orientation`.

Логика:

1. Если позиция живёт дольше `position_ttl`, бот переводит цель в TTL-close режим.
2. Цель должна ставиться относительно точки входа с учётом `to_entry_orientation`.

   * `0.0` = ровно entry price;
   * положительное/отрицательное значение = смещение относительно entry. -- задокументировать.
3. Если TTL-close не исполняется за дополнительное окно ожидания, включается extrime mode.

---

## 1.10.5. Extrime close

Назначение: быстро закрыть позицию лимитками, ухудшая качество выхода постепенно, но без маркета.

Параметры:

* `retry_ttl`;
* `retry_num`;
* `bid_to_ask_orientation`;
* `increase_fraction`.

Логика:

1. Каждые `retry_ttl` секунд вычисляется новая exit-цена.
2. Цена рассчитывается:

   * относительно mid = `(ask1 + bid1) / 2`; -- ось.
   * bid_to_ask_orientation == 0 -- прямо в эту ось, с последующими смещениями в компромисную сторону.
   * с нарастающим ухудшением через `increase_fraction`. increase_fraction -- % от (ask1 - bid1). (ask1 - bid1) == 100%.
3. Если число попыток ограничено и лимит исчерпан:

   * действие не ожидается. Молимся Богу.

# 1.11. Завершение позиции

Позиция считается завершённой, если:

* закрывающий ордер исполнился;
* либо биржа сообщает, что размер позиции стал 0;
* либо оператор вручную закрыл позицию и биржа это подтвердила.

После завершения:

* позиция удаляется из active state;
* pending entry/interference снимаются;
* close_order_id очищается;
* state сохраняется.

Если закрытие было обычным торговым результатом:

* обновляется fail/success статистика.

Если закрытие было внешним/ручным:

* только reset state;
* **без штрафов, без карантина, без повторного входа по самовольству бота**.

---

# 1.12. Recovery после рестарта

При старте бот обязан:

1. загрузить локальный state;
2. спросить биржу о реальных открытых позициях;
3. выполнить reconciliation:
   * если локальная позиция есть и биржевая есть — восстановить;
   * если локальная есть, а биржевой нет — удалить локальную;
   Точка.
4. спросить активные ордера;
5. сопоставить известные TP/entry/interference ордера;
6. Никакого привития диких маслин! (Инвариант уже прописал ранее).

# 1.15. Нефункциональные требования

## Производительность

* бот должен выдерживать поток стаканов по множеству символов без накопления бесконечной очереди задач;
* обработка по одному символу должна быть последовательной. Примечание: если процессорные задачи то да. Если зависят от O/I то асинхронно.

## Устойчивость

* дубли WS-событий не должны ломать логику;
* реконнекты должны быть прозрачны;
* state должен переживать рестарты.

## Целостность

* один и тот же ордер не должен интерпретироваться как два разных события;
* один и тот же символ не должен одновременно находиться в двух торговых циклах (Наверное инвариант одновременной торговли по смежной стороне (если был сигнал в противоположную сторону за время работы со стороной А)) (Но со строгими границами ответственности per pos_side). Такой Хедж-Мод бота должен выбираться флагом в конфигах в разделе risk.

## Progress / 2026-03-30

Проведен повторный аудит проекта относительно текущего ТЗ. Подтверждены и закрыты реальные расхождения в critical-логике жизненного цикла позиции.

Что исправлено:

1. Частичный вход теперь не блокирует сопровождение позиции.
   - В state позиция больше не создается с `qty = плановый объем ордера`.
   - До первого fill `qty = 0`, а `init_qty` хранит плановый объем.
   - После partial fill бот начинает работать по реально налитому объему сразу, не дожидаясь финального cancel-ack.

2. Отмена хвоста входной лимитки вынесена в фоновый асинхронный путь.
   - Повторные `PartiallyFilled` не дублируют cancel.
   - Первичная TP при `average.stabilization_ttl == 0` может быть поставлена параллельно отмене хвоста.

3. Тот же принцип применен к скупке помех.
   - Если ордер помехи исполнен частично, хвост снимается асинхронно.
   - Дальше бот работает только с реально долитым объемом.

4. Внешнее/ручное закрытие больше не идет через PnL/fail/quarantine ветку.
   - Выполняется только reset state + cancel_all_orders(symbol) + save.
   - Fail counter и quarantine на manual/external close не трогаются.

5. `average.enable = false` больше не приводит к скрытой первичной постановке TP.
   - Bootstrap INITIAL_TP теперь уважает флаг `average.enable`.

6. `position_ttl_close.to_entry_orientation` закреплен в явной TTL-логике.
   - Расчет TTL-target вынесен в отдельный метод `build_target_price()`.
   - Exit engine использует уже готовую цену, а не декоративное поле.

7. `stop()` переведен в корректный graceful shutdown.
