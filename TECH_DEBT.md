# ТЕХНИЧЕСКОЕ ЗАДАНИЕ И АРХИТЕКТУРНЫЙ ДОЛГ БОТА
**Статус ядра:** [x] Стабильно | **Статус WS:** [x] Стабильно | **Hedge Mode:** [x] Имплементирован

## 1. Назначение
Разработать HFT асинхронного торгового бота для **Phemex Futures**. Лимитная логика входа, динамическая лимитная логика выхода ("снайпер"), строгий контроль стейта через вебсокеты. Бот работает с ликвидностью напрямую, избегая рыночных (Market) ордеров.

## 2. Основной режим и Конфигурация
* Биржа: Phemex USDT Futures.
* Мультиактивность: Контроль слотов через `app.max_active_positions >= 1` (лимит считается по количеству уникальных символов в работе).
* Hedge-mode (концептуальный): Допускается одновременное нахождение в Long и Short по одной монете (если разрешено конфигом). Изоляция гарантируется композитными ключами (например, `BTCUSDT_LONG`).

## 3. Золотые Инварианты Системы
1. **Изоляция:** По одной монете-стороне не может быть двух циклов.
2. **Лимитная природа:** Входы и выходы осуществляются строго лимитными ордерами.
3. **Зачистка остатков:** Любое частичное исполнение (`PartiallyFilled`) приводит к немедленной фоновой отмене остатка ордера через WS, расчищая стейт для новой оценки стакана.
4. **Сброс по инициативе юзера:** При ручном закрытии позиции оператором или ликвидацией стейт обнуляется штатно без карантинов.
5. **Сетевой отказ (Reject):** Если бот находится в позиции, стейт **никогда не удаляется** при ошибках API. При исчерпании лимита ошибок позиция переходит в Экстрим-режим, далее — блокировка и вызов оператора.

## 4. Жизненный цикл выхода (Exit Lifecycle)

### 4.1. Average Scenario & Фаза Охоты (Hunting Phase)
Бот вычисляет виртуальный Тейк Профит на базе `ask2/bid2` момента сигнала. При достижении условий (поиск достаточного объема в стакане >= или <= виртуальной цели) бот совершает **физический выстрел (удар по стакану)**.
* **Фаза Охоты:** При выставлении ордера позиция блокируется на `hunting_timeout_sec` (по умолчанию 1 сек). В это время таймеры деградации (снижения) цели замораживаются.
* **Развязки Охоты (Третьего не дано):**
  1. Полное исполнение -> Стейт обнуляется, торговый цикл завершен.
  2. Частичное исполнение -> Остаток отменяется мгновенно (работает инвариант №3).
  3. Таймаут Охоты истек -> Снимается блокировка. На следующем тике стакана цель может быть пересмотрена (деградирована). Если цель меняется, старый ордер принудительно отменяется перед постановкой нового.

### 4.2. Negative Scenario
Если позиция болтается в негативной зоне (спред хуже цены входа) на протяжении `negative_ttl`, позиция признается неудачной и передается в блок Экстрим.

### 4.3. Interference (Скупка помех)
Скупка мелких объемов, мешающих дойти до основной цели (аски для лонга, биды для шорта). Зона сканирования: между ценой входа и виртуальной целью. Купленный объем плюсуется к балансу позиции. Благодаря тому, что физический тейк-профит не висит в стакане статично, Average Scenario на следующем тике просто учтет увеличенный объем при расчете нужной ликвидности для выхода.

### 4.4. Position TTL
Жесткий таймер жизни позиции. При истечении цель принудительно переводится в безубыток (+ `to_entry_orientation`). При неисполнении — переход в Экстрим.

### 4.5. Extrime Close
Последний рубеж (Emergency Exit). Вычисляет спред-ось `mid = (ask1 + bid1) / 2` и выставляет лимитный ордер. Каждые `retry_ttl` секунд ордер переставляется с прогрессирующим смещением `shift` (агрессивно пробивая стакан вглубь), эмулируя маркет-ордер, но сохраняя лимитную защиту от проскальзывания.

## 5. Recovery (Холодный старт)
При старте бот поднимает локальный `BotState` и делает REST-запрос позиций. 
* **Амнезия:** Все забытые локально (но висящие на бирже) позиции игнорируются (нет "удочерения").
* **Зачистка:** При несовпадении объемов бот синхронизируется с биржей, принудительно снося все зависшие на бирже лимитные ордера по данному символу, чтобы начать работу с "чистого листа".
