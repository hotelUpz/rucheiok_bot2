## GOAL

# 1. Полное ТЗ бота, как будто проекта ещё нет

## 1.1. Назначение

Разработать асинхронного торгового бота для **Phemex Futures**, работающего по лимитной логике входа и лимитной логике выхода, с контролем позиции по стакану и ценовым условиям.

Бот должен:

* находить точку входа по паттерну стакана;
* открывать позицию **только лимитным ордером**;
* после подтверждения входа сразу переводить позицию в режим управляемого выхода;
* сопровождать позицию несколькими параллельными exit-сценариями;
* работать устойчиво при частичных исполнениях, отменах, ручных действиях оператора, реконнектах WS и рестартах процесса.

Сразу правило частичных исполнений как открывающих лимиток так и закрывающих: как только от биржи пришло подтверждение о частичном исполнении ордера -- остаток отменяем!

---

## 1.2. Основной режим работы

* биржа: **Phemex USDT Futures**;
* архитектура должна поддерживать торговлю `N` > 1 позициями (режим мультиактивов, то есть параллельно).

Конфигурация:

* `app.max_active_positions = 1` по умолчанию;
* app.max_active_positions > 1 -- режим мультиактивов.

---

## 1.3. Источники входных данных

Бот получает данные из следующих источников:

### Публичные данные

* Phemex order book / depth stream;
* Phemex тикеры/цены;
* Phemex funding;
* Binance цены для контр-проверки спреда.

  * обновления позиций;
  * факты fill / partial fill / cancel / reject.

Любому статусу reject даем max_place_order_retries попыток перестановки.

### Локальные входы

* `cfg.json`;
* `.env`;
* Telegram-команды оператора;
* локально сохранённый state.

## 1.7. Правила торговли

### Инварианты

1. По одной монете не может быть двух независимых торговых циклов одновременно. (за исключением если речь идет о противоположной стороне. Хедж мод режим должен быть прописан в конфигах). Хедж мод имеется в виду в первую очередь концептуальный, не биржевой, что одновременно может быть две позиции если по второй позиции (по Шорту например) пришел сигнал. Все ключи доступа должны быть per symbol + per position_side.
2. Пока позиция не завершена, монета-сторонаПозиция считается занятой.
3. Все входы — только лимиткой.
4. Все выходы — только лимиткой.
5. Остаток входной лимитки после partial fill обязан быть снят. (В фоновом режиме, не блокируя выполнения остальных операций, например -- постановку закрывающей TP лимитки. Важно!). То же правило относится и к закрывающим лимиткам.
6. После внешнего/ручного закрытия бот не “додумывает” поведение за оператора:

   * нет позиции → state сброшен;
   * монета-сторона снова свободна;
   * без штрафов, без карантина, без искусственных кулдаунов.
7. Решения по входу и выходу принимаются только по свежим данным.
8. Любое действие бота должно быть идемпотентным: повтор одного и того же WS-события не должен ломать логику.
9. Источник для входа в позицию -- только внутренние сигналы. Никаких удочерений осиротевших позиций. Благотворительностью не занимаемся и не лезем в чужие торги.

# 1.8. Подробная логика входа

## 1.8.1. Universe

Бот получает список активных USDT-фьючерсных контрактов Phemex.

Далее исключаются:

* black list;
* неактивные символы;
* символы в карантине;
* символы уже занятые стороной-позицией/ордером.

---

## 1.8.2. Entry pattern

По стакану Phemex анализируется паттерн:

* верхняя часть;
* тело;
* низ стакана;
* соотношение уровней;
* допустимая дистанция bid/ask;
* глубина анализа.

Результат:

* либо сигнала нет;
* либо сигнал есть и содержит:

  * position side;
  * entry price;
  * параметры для расчёта цели. (по сути entry price и количество позиции)

---

## 1.8.3. Binance cross-check

Если включён Binance filter:

* сравниваются Binance и Phemex цены;
* проверяется минимальный спред;
* при необходимости используется TTL подтверждения спреда.

---

## 1.8.4. Funding filter

Если включён funding filter:

* символ скипается перед funding-событием;
* символ скипается, если funding превышает порог.

---

## 1.8.5. Финальное условие входа

Вход возможен только если одновременно:

* есть валидный pattern signal;
* пройден Binance-filter;
* символ прошел funding filter;
* символ не в black list;
* символ не в карантине;
* символ не занят конкретной стороной (например Лонг и по Лонгу сигнал);
* не превышен лимит активных позиций. В хедж моде 1 лимит это Лонг + Шорт по одному символу.

---

## 1.8.6. Расчёт размера входа

Размер позиции определяется по `risk`:

* `margin_size = row` или фиксированное значение; Если 'row' -- берем из первого уровня аска (для Лонг сценария) и первого уровня бида (для Шорт сценария). Суть -- выкупка объема первого уровня стакана.
* берётся объём первого уровня стакана;
* добавляется `margin_over_size_pct`; (С целью запаса на случай появления дополнительного объема в момент детерминации сигнала)
* ограничивается `notional_limit`; -- святое правило.
* переводится в количество контракта;
* округляется по `lot_size`. -- золотой стандарт вычисления.

---

## 1.8.7. Размещение входного ордера

Бот выставляет лимитный ENTRY order:

* side = Buy/Sell; (Лучше перейти на семантику Хедж Режима: Open Long / Open Short, Close Long / Close Short)
* pos_side = Long/Short;
* qty;
* price.

После успешной постановки:

* создаётся position state;
* сохраняется `pending_entry_order`;
* позиция считается “открывающейся”, но ещё не подтверждённой.

---

1.8.8. Обработка входного ордера по WS

Если ордер исполнен полностью — вход завершён, запускается сопровождение позиции.
Если ордер исполнен частично:
фиксируем фактически исполнённый объём;
отправляем асинхронную отмену остатка;
дальше работаем с фактически исполнённым объёмом;
Если ордер отменён без исполнения (например руками):
удаляем временный state;
освобождаем символ-сторону.
Если ордер отклонён или деактивирован до входа:
удаляем временный state;
освобождаем символ-сторону;
(fail/quarantine не начисляем.)

---

# 1.9. Запуск exit-жизненного цикла

После подтверждения входа должны параллельно запуститься механизмы управления позицией:

* average scenario;
* negative scenario;
* interference monitor;
* global position TTL;
* extrime close engine.

Важно:

* average и negative и interference блоки стартуют **одновременно** и живут параллельно;
* TTL является глобальным ограничением жизни позиции;
* extrime включается только по условиям (см ниже).

---

# 1.10. Подробная логика выхода

## 1.10.1. Average scenario (Сердце торговой стратегии)

(Без правильной постановки нижесказанного, система просто не будет работать!!)

Параметры:

* `stabilization_ttl`;
* `target_rate`;
* `shift_demotion`;
* `min_target_rate`;
* `shift_ttl`.

Поведение:

1. После подтверждения входа ждём `stabilization_ttl`. (0 -- не ждем)
2. После ожидания вычисляем первичную TP:

   * цена определяется как доля пути от `entry_price` до `base_target_price_100`.
   sign = 1 if pos_side.upper() == "LONG" else -1
   Формула = entry_price(то есть ask1 | bid1) * (1 + sign *(spr2_pct * target_rate))
   то есть base_target_price_100 de facto == ask2 для Лонг (на момент сигнала!) и bid2 для Short (на момент сигнала!) -- нужно добавить в стейт для расчетной базы. ask2 и bid2 берем из StakanEntryPattern на момент сигнала.
   Значение этой найденной цены округляем в соответствии правилами тиков стакана (по сути ценовой округлитель). Это очень важно! Либо другой подъод: когда будем например для Лонгов парсить биды (искать точку выхода), то просто вычисляем процент между текущими бидами и нашей точкой входа. Понятно что он должен быть положительным и >= (spr2_pct * target_rate)Еще. если в стакане дают лучшего бида на продажу (то есть для лонга, повыше, то кидаем лимитку именно в него)

3. Главная магия: найденное значение мы не просто ставим лимитным ордером на закрытие, а:
   Для Лонга: ищем наличие объема в бидах(!) по интересующей нас цене.
   Для Шорта: ищем наличие объема в асках(!) по интересующей нас цене.
   Внимание, фокус: как только засекли такой объем по интересующей нас цене -- кидаем туда лимитку на закрытие. (Интересующая нас цена -- смотри пункт выше). Выполняем это динамическое действие в течение 1 минуты (конкретно shift_ttl sec (из конфигов)). Если закроется поза целиком -- отлично. Фиксируем итерацию, обновляем стейт и идем дальше. Бот теперь снова отзывчив к новым сигналам по этой стороне (и по всем незанятым (актуальная реплика в контексте хедж режима)).

4. Выкупаем любой объем. Остаток -- сразу отменяем.

5. Если в течение окна shift_ttl позиция все еще открыта (остался какой либо остаток) тогда идем на компромис и -- перевычисляем цель по формуле 2. Но уже от target_rate минусуем shift_demotion. 

6. Если остался остаток меньше min_notional -- оставляем позицию и считаем ее неактивной. Кеш и стейт обновляем.
(где то захардкоден как 5 баксов но лучше прописывать в конфигах, а еще лучше -- брать из спецификации по конкретной монете).  

8. Когда в результате перестановок если дошли до min_target_rate (и подождали еще минуту, то есть попытались отработать этот уровень (точнее не минуту а снова таки shift_ttl)), то переходим к сценарию extrime_close закрытия.

8. После каждого сдвига и каждой отработке: negative_duration сбрасывается.

Если `average.enable == false`, этот сценарий не должен вообще управлять целью. Однако это сердце торговой стратегии.

## 1.10.2. Negative scenario

Назначение: определить, что рынок после входа пошёл против позиции (либо стоит на месте), и через выдержанный негативный период перевести позицию в экстренное закрытие.

Параметры:

* `stabilization_ttl`;
* `negative_spread_pct`;
* `negative_ttl`.

Логика:

1. До истечения `stabilization_ttl` сценарий молчит.
2. Для LONG:

   * сравнивается текущий ask1 с init_ask1.
3. Для SHORT:

   * сравнивается текущий bid1 с init_bid1.
4. Если рынок находится в негативной зоне (то есть например ask1 == init_ask1 or bid1 == init_bid1 (при negative_spread_pct == 0), либо в худшую сторону):

   * идёт накопление negative_duration.
5. Если negative_duration >= negative_ttl:

   * включается extrime mode.

Если рынок перестал быть негативным (то есть этот сценарий стакана):

* negative_duration обнуляется.

---

## 1.10.3. Interference

Назначение: скупка мешающих уровней между входом и текущей целью.
Важно: для Лонга: такие помехи будут в асках (искать и скупать только там)
Важно: для Шорта: такие помехи будут в бидах (искать и скупать только там)

Параметры:

* `average_vol_pct_to_init_size`; -- если объем в пределах процента average_vol_pct_to_init_size от init_size (начально планового объема перед постановкой открывающей лимитки), то такое выкупаем.
* `max_vol_pct_to_init_size`. -- суммарное значение скупок которые бот может себе позволить (как процент max_vol_pct_to_init_size от init_size (-- то есть начально планового объема перед постановкой открывающей лимитки))

Логика:

* Для LONG анализируются asks между `entry_price` и плановым виртуальным ТЕЙК ПРОФИТОМ!.
* Для SHORT анализируются bids между плановым виртуальным ТЕЙК ПРОФИТОМ и `entry_price`.

Докупка приурочивается к общему объему позиции и бот ее пытается сбагрить по такому же принципу как и основной объем (смотри раздел '1.10.1. Average scenario (Сердце торговой стратегии)')

---

## 1.10.4. Global position TTL

Назначение: позиция не должна жить бесконечно.

Параметры:

* `position_ttl`;
* `to_entry_orientation`.

Логика:

1. Если позиция живёт дольше `position_ttl`, бот переводит цель в TTL-close режим.
2. Цель должна ставиться относительно точки входа с учётом `to_entry_orientation`.

   * `0.0` = ровно entry price;
   * Тип данных: +float (или 0). Если > 0 то ставим в худшую сторону на величину этого процента от точки входа.
3. Если TTL-close не исполняется за дополнительное окно ожидания, включается extrime mode.

---

## 1.10.5. Extrime close

Назначение: быстро закрыть позицию лимитками, ухудшая качество выхода постепенно, но без маркета.

Параметры:

* `retry_ttl`;
* `retry_num`;
* `bid_to_ask_orientation`;
* `increase_fraction`.

Логика:

1. Каждые `retry_ttl` секунд вычисляется новая exit-цена.
2. Цена рассчитывается:

   * относительно mid = `(ask1 + bid1) / 2`; -- ось.
   * если bid_to_ask_orientation == 0 -- прямо в эту ось, с последующими смещениями в компромисную сторону.
   * с нарастающим ухудшением через `increase_fraction`. increase_fraction -- % от (ask1 - bid1). (ask1 - bid1) == 100%.
3. Если число попыток ограничено и лимит исчерпан:

   * Ничего не делаем. Но по концепту будет стоять "inf", то есть постепенно будем смещаться вплоть до закрытия.

# 1.11. Завершение позиции

Позиция считается завершённой, если:

* закрывающий ордер целиком исполнился;
* либо биржа сообщает, что размер позиции стал 0;
* либо оператор вручную закрыл позицию и биржа это подтвердила.

если Размер позиции < min_notional то это равносильно 0 количеству.

После завершения:

* позиция удаляется из active state;
* pending entry/interference снимаются;
* close_order_id очищается;
* state сохраняется.
* state приходит в первозданное состояние.

Если закрытие было обычным торговым результатом:

* обновляется fail/success статистика.

Если закрытие было внешним/ручным:

* только reset state;
* **без штрафов, без карантина, без повторного входа по самовольству бота**.

---

Правило карантина:
если количество проигрышей по позиции превысило max_consecutive_fails то заносим в карантин на quarantine_hours часов (либо сразу в черный список (если quarantine_hours == "inf"))

# 1.12. Recovery после рестарта

При старте бот обязан:

1. загрузить локальный state;
2. спросить биржу о реальных открытых позициях;
4. спросить активные ордера;
5. сопоставить известные TP/entry/interference ордера;
3. выполнить reconciliation:
   * если локальная позиция есть и биржевая есть — восстановить и возобновить торговый алгоритм;
   * если локальная есть, а биржевой нет — удалить локальную;
   Точка.
6. Никакого привития диких маслин! (Инвариант уже прописал ранее).
